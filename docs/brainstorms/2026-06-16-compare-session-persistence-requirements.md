## Резюме

Хранить выбор модулей для сравнения в сессии Django вместо одноразовых
GET-параметров `/compare/?ids=...`. Чекбоксы в каталоге сразу обновляют
список в сессии, поэтому выбор «переживает» переходы по страницам каталога
и сохраняется между визитами (как и сейчас живёт сессия — около двух
недель). Доступ к списку — через постоянный значок в навигационной панели.
Новая модель или таблица в БД не добавляется: данные используют встроенный
механизм сессий Django (служебная таблица `django_session`).

## Контекст проблемы

Сейчас выбор для сравнения — это чекбоксы в форме каталога
(`templates/products/product_list.html`), которые отправляются GET-запросом
на `/compare/?ids=1&ids=2...` (`products/views.py:168-196`). Это состояние
существует только в момент отправки формы: при переходе на другую страницу
пагинации, изменении фильтров, открытии карточки товара и возврате назад —
все отметки сбрасываются. Пользователь не может постепенно собрать список из
2-4 модулей, просматривая разные страницы каталога, и не может вернуться к
ранее собранному сравнению в другой день.

## Ключевые решения

- **Хранилище — сессия Django, без новой модели.** `SessionMiddleware` уже
  включён (`config/settings.py`), и сессия по умолчанию переживает закрытие
  браузера (`SESSION_EXPIRE_AT_BROWSER_CLOSE=False` по умолчанию, срок жизни
  около двух недель). Список id хранится в `request.session` и физически
  попадает в служебную таблицу `django_session` — она управляется самим
  Django и не считается новой таблицей проекта.
- **Срок жизни — без изменений.** `SESSION_COOKIE_AGE` не меняется: это
  глобальная настройка всего проекта, влияющая на все сессии, а не только на
  сравнение.
- **Мгновенное обновление при клике на чекбокс.** Список в сессии
  обновляется сразу при изменении чекбокса в каталоге, а не только при
  отправке формы «Сравнить», чтобы выбор накапливался по мере просмотра
  страниц.
- **Постоянный доступ к сравнению с любой страницы.** В навигационной панели
  появляется ссылка/значок «Сравнение (N)», ведущая на `/compare/`, видимая
  независимо от текущей страницы.
- **`/compare/` читает список только из сессии.** Параметр `?ids=` перестаёт
  быть источником данных для страницы сравнения; лимит 2-4 модуля сохраняется
  как и сейчас.

## Требования

**Хранение выбора**

- R1. Выбор модулей для сравнения хранится в сессии Django (список id, не
  более 4), а не передаётся через GET-параметры.
- R2. При изменении состояния чекбокса в каталоге список в сессии
  обновляется немедленно, без отправки формы «Сравнить».
- R3. Если в списке уже 4 модуля, добавление пятого не сохраняется в сессии;
  UI отражает фактический список (как и сегодня, лишние чекбоксы блокируются,
  но проверка дополнительно выполняется при обновлении сессии).
- R4. При открытии страницы каталога — включая переход на другую страницу
  пагинации, смену фильтров или возврат с карточки товара — чекбоксы ранее
  выбранных модулей отображаются как отмеченные.

**Доступ и просмотр**

- R5. В навигационной панели (`templates/base.html`) отображается
  ссылка/значок с текущим количеством выбранных модулей, ведущая на
  `/compare/`, видимая на любой странице сайта.
- R6. Страница `/compare/` отображает модули из списка, сохранённого в
  сессии (а не из параметра `?ids=`).
- R7. На странице `/compare/` есть действие «Очистить список», полностью
  обнуляющее сохранённый выбор.

**Устойчивость данных**

- R8. Если модуль, сохранённый в списке сравнения, больше не существует в БД
  (например, удалён), он не отображается на `/compare/` и не учитывается в
  счётчике значка — без ошибок.

## Примеры приемки

- AE1. **Покрывает R2, R4.** Пользователь отмечает 2 модуля на странице 1,
  переходит на страницу 2 каталога — отметки со страницы 1 сохраняются
  (видны в значке навбара, восстанавливаются при возврате на страницу 1).
- AE2. **Покрывает R3.** В списке сравнения уже 4 модуля. Попытка отметить
  пятый чекбокс не добавляет его в сессию; UI продолжает показывать список из
  4 модулей.
- AE3. **Покрывает R5, R6, R7.** Пользователь выбрал 3 модуля в каталоге,
  переходит по ссылке «Сравнение (3)» из навбара на любой странице сайта →
  видит таблицу сравнения этих 3 модулей. Нажимает «Очистить список» → список
  становится пустым, значок показывает 0.
- AE4. **Покрывает R8.** В сессии сохранён id модуля, который был удалён из БД
  между визитами. При открытии `/compare/` этот модуль не отображается,
  счётчик в навбаре отражает только существующие модули.

## Границы охвата

**Вне рамок**

- Изменение `SESSION_COOKIE_AGE` или иных глобальных настроек сессий — срок
  жизни сравнения наследует текущие настройки сессии проекта.
- Раздельное хранение списка сравнения для разных пользователей — аккаунтов в
  проекте нет, разделение не требуется.
- Синхронизация списка сравнения между разными браузерами/устройствами одного
  человека — список привязан к cookie сессии конкретного браузера.
- Прямые ссылки на сравнение вида `/compare/?ids=1&ids=2` — после перехода на
  сессию такие ссылки не формируют список сравнения.

## Зависимости / допущения

- `django.contrib.sessions` и `SessionMiddleware` остаются включёнными в
  `config/settings.py` (уже так, без изменений).
- Чекбоксы и кнопка «Сравнить» в `templates/products/product_list.html`
  (строки 90-137, 144-166) и представление `compare()` в `products/views.py`
  (строки 168-196) требуют переработки: обновление сессии при изменении
  чекбокса вместо накопления значений только в форме, и чтение списка модулей
  из сессии вместо `request.GET.getlist('ids')`.
